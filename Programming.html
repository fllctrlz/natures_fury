Programming

  • We use math blocks, variables, logic, compares, loops switches, and MyBlocks with data hubs (so we could reuse code but change parameters)
  • We have created a selection program which allows us to quickly and efficiently choose missions (or even re-run missions quickly) during matches. Audible sounds and meaningful NXT screen messages allow the user to select programs with ease
  • By using gear lash removal before each movement, the ramp up feature when starting from stationary position, and our own error-corrected move block, we are able to achieve more consistent and repeatable robot movement
  • We regularly use subroutines (i.e. MyBlocks) to reuse code whenever possible and to keep our program sizes manageable. See diagram of our NXT-G programming hierarchy:



  • Our MyBlocks and variables are given descriptive and meaningful names to allow us to easily understand what our programs are doing
  • We annotate our work to help us debug programs and to better understand each programming block's purpose



  • We program our robot to drive along walls and square to mission models, walls, and lines to maximize repeatability and minimize the likelihood that our team will incur a touch penalty
  • We program the robot to use 2 light sensors for navigational flexibility and rotation sensors to measure distance
  • We make frequent use of our square up subroutine which allows the robot to position itself perpendicularly to any black line (from virtually any angle of attack)
  • We designed our own error-corrected move block subroutine for straighter and more accurate movement. It constantly compares the distances travelled by each motor and adjusts each's speed accordingly to maintain the same ratio between the distance travelled by each motor. The subroutine can be used for accurate arc turns.
  • When accelerating from a stationary position, our robot uses a Ctrl-Z-designed ramp up feature to minimize the likelihood of wheel slip. This provides for greater repeatability of the robot's movements
  • We designed our own PID (Proportional, Integral, Derivative) subroutine that allows for smoother and faster line following than the standard 2-stage line following. It calculates the difference between the current light sensor reading and the optimal reading (half black and half white)
  • When we measure light readings, we use raw values instead of calibrated values. Using raw values has two distinct advantages: a) it has double the granularity (calibrated values have a range of 0 to 100 while raw values have a range of about 450 to 650), and b) it allows the robot to calibrate for each light sensor (this is important because light sensors do not necessarily return the same values for a given light intensity)
  • When we first programmed the mission that performs tasks on the East side of the playing field, we tried "dead reckoning", or navigating without landmarks, to the Northeast corner due to a lack of distinguishable reference points. However, after testing the program multiple times, we found that it was unrepeatable. We then designed a program that determines the position of one of the robot's light sensors on the blue "F" of the FLL logo located in the centre of the mat by measuring the distance travelled from base to the blue "F". Based on the robot's position on the "F", the program decides how much the robot should pivot so that it arrives consistently at the Northeast corner of the playing field.
